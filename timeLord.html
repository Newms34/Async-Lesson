Time Complexity

Time complexity is a complex (hah!) subject, but here's the basic gist:

  1. Take some sort of operation. For this example, I'll pretend I'm asking my friends Bob, Bill, Betty, and Bianca to a party.
  2. Let's assume that I can only set a crucial detail of the party - how much pizza I need to order - after I know everyone's responses.
  3. We saw in Async two ways to do this:
    • Callback Hell
    • Promises
  4. But there's a THIRD way!:

  5. So, what's the issue here? Well, consider that for any number of organisms n, we need to run n2 loop iterations. That's gonna end up being a large number, very quickly. From wikipedia: "the amount of time taken by an algorithm to run as a function of the length of the [...] input"
  6. When we're programming, we want to aim for the lowest time complexity possible. Some examples of different complexities:
    • O(1): Time to complete operation does not change depending on the input. Assuming the basal function is quick, this is usualy ideal (but not always possible!)
    • O(n): For any number of inputs n, the time to complete the operation increases linearly. So if 1 item takes 1 second, and 2 items take 2 seconds, then 3 items takes 3 seconds, and so on.
    • O(log(n)): For any number of inputs n, the time to completion increases by the logarithm of n.
    • O(n2): For any number of inputs n, the time to completion increases by the square of the input. My beasties 🐆 example above is O(n2).
    • O(kn) (where k is some constant): For any number of inputs n, the time to completion increases by some number k raised to that number. This kinda function increases time complexity very quickly, so it's very undesirable (unless there are no other reasonable choices!).
